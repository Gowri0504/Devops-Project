4. ARCHITECTURE DEEP-DIVE (THE "HOW")
-------------------------------------
We implement a "Defense in Depth" strategy using 3 distinct tiers:

TIER 1: THE PUBLIC GATEWAY (Frontend)
- Uses Nginx to serve static React files.
- Resides only in 'app-network'.
- Has NO direct access to the database.

TIER 2: THE SECURE BRIDGE (Backend)
- Handles authentication, logic, and database queries.
- Resides in BOTH 'app-network' and 'db-network'.
- Acts as the only authorized "hop" between users and data.

TIER 3: THE ISOLATED VAULT (Database)
- MongoDB instance holding all student records.
- Resides only in 'db-network'.
- Completely invisible to the public internet and Tier 1.

6. NETWORKING STRATEGY (VPC-LIKE ISOLATION)
-------------------------------------------
Why use multiple networks?
In a standard Docker setup, all containers join a single bridge network. This means if your
frontend is hacked, the hacker can directly attack your database.

In OUR setup:
- app-network: `172.18.0.0/16` (Sample subnet)
- db-network: `172.19.0.0/16` (Sample subnet)

The 'frontend' and 'db' are on different subnets with no routing between them. Only the
'backend' has two "NICs" (Network Interface Cards), allowing it to talk to both. This is
exactly how professional VPCs (Virtual Private Clouds) are architected in AWS or Azure.

7. DOCKERFILE & COMPOSE BREAKDOWN
---------------------------------
Dockerfile.frontend (Multi-stage):
- Stage 1 (Builder): Uses Node to compile React into static HTML/JS/CSS.
- Stage 2 (Runner): Uses Nginx to serve those files.
- BENEFIT: The final image doesn't contain Node.js or source code, making it tiny and secure.

docker-compose.yml:
- `depends_on`: Ensures DB starts before Backend, and Backend starts before Frontend.
- `restart: unless-stopped`: Provides self-healing if a container crashes.
- `volumes`: Ensures data persistence.

8. SECURITY HARDENING & BEST PRACTICES
--------------------------------------
- Alpine Linux: We use 'alpine' variants for all images to reduce the attack surface.
- Internal DNS: We use container names (backend, db) instead of IP addresses.
- No Host Ports: The database port 27017 is NOT exposed to your laptop. It's only 
  open INSIDE the db-network.
- Reverse Proxy: Nginx hides the fact that we are using Node.js on port 3000.

9. THE FULL DATA WORKFLOW (REQUEST TO RESPONSE)
-----------------------------------------------
1. User enters data on Dashboard.
2. Frontend sends a request to `http://localhost/api/topics`.
3. Nginx receives it on port 80.
4. Nginx looks at the `/api` part and says "I need to send this to the backend container".
5. Request travels through `app-network` to `backend:3000`.
6. Backend processes the logic and queries `db:27017` via `db-network`.
7. MongoDB returns data to Backend.
8. Backend sends JSON back to Nginx.
9. Nginx sends it back to the User's browser.

1. Network Segmentation: By separating the database into its own network, we prevent direct access from the public internet.
2. Minimal Images: Using Alpine Linux reduces the number of pre-installed tools that an attacker could exploit.
