4. DETAILED DEVOPS ARCHITECTURE (THE "HOW")
-------------------------------------------

A. MULTI-STAGE DOCKER BUILDS
   - Location: [Dockerfile.frontend](file:///d:/DEV/Project-Docker/track-placement/Dockerfile.frontend)
   - Technique: We use a 2-stage build. 
     - Stage 1: Build the React app using Node:18-alpine.
     - Stage 2: Transfer ONLY the 'dist' folder to an Nginx:alpine image.
   - Why: This reduces the final image size from ~400MB to ~20MB, improving security 
     by removing build tools and increasing deployment speed.

B. NETWORK SEGMENTATION (VPC-LIKE ISOLATION)
   - Location: [docker-compose.yml](file:///d:/DEV/Project-Docker/track-placement/docker-compose.yml)
   - Networks:
     1. `app-network`: Connects the Frontend (Nginx) and Backend (Node.js).
     2. `db-network`: Connects the Backend (Node.js) and Database (MongoDB).
   - Why: This mimics a real cloud VPC. The Database is on a private network and 
     CANNOT be reached by the frontend or the public internet. If the frontend is 
     hacked, the database remains safe.

C. NGINX REVERSE PROXY & API GATEWAY
   - Location: [nginx.conf](file:///d:/DEV/Project-Docker/track-placement/nginx.conf)
   - Implementation: Nginx listens on Port 80. It serves the HTML files and 
     proxies all requests starting with `/api/` to `http://backend:3000/api/`.
   - Why: 
     - Prevents CORS (Cross-Origin Resource Sharing) errors.
     - Hides the backend server identity (Security through obscurity).
     - Single point of entry for the entire system.

D. PERSISTENT STORAGE (DOCKER VOLUMES)
   - Location: [docker-compose.yml](file:///d:/DEV/Project-Docker/track-placement/docker-compose.yml)
   - Implementation: `db-data:/data/db` maps a local folder to the MongoDB data directory.
   - Why: Containers are "ephemeral" (they die and lose data). Volumes ensure 
     that student placement data survives container updates, crashes, or deletions.

E. DYNAMIC SERVICE DISCOVERY
   - Technique: Docker's internal DNS.
   - Usage: In [nginx.conf](file:///d:/DEV/Project-Docker/track-placement/nginx.conf), we use `proxy_pass http://backend:3000`. 
   - Why: We don't need hardcoded IP addresses. Docker handles the routing 
     automatically using the container name.

21. DOCKER NETWORKING STRATEGY (SPECIFIC TO REQ 36)
---------------------------------------------------
This project implements the "Docker Networking with Isolated Containers" 
requirement by:
- CREATING CUSTOM NETWORKS: We define `app-network` and `db-network` as bridge 
  networks in [docker-compose.yml](file:///d:/DEV/Project-Docker/track-placement/docker-compose.yml).
- RUNNING CONTAINERS IN ISOLATED NETWORKS:
  - `frontend` is in `app-network` only.
  - `db` is in `db-network` only.
  - `backend` is the bridge, in BOTH networks.
- ENABLING COMMUNICATION BETWEEN SELECTED CONTAINERS:
  - `frontend` <-> `backend` can talk on `app-network`.
  - `backend` <-> `db` can talk on `db-network`.
- TESTING NETWORK ISOLATION:
  - We provide [test_isolation.sh](file:///d:/DEV/Project-Docker/track-placement/test_isolation.sh) 
    to demonstrate that `frontend` CANNOT ping or reach `db`.

22. WORKFLOW OF THE NETWORK ISOLATION TEST
------------------------------------------
1. RUN SCRIPT: Execute `bash test_isolation.sh`.
2. BACKEND -> DB: The script enters the `backend` container and pings the `db` 
   container. Result: SUCCESS (Controlled communication enabled).
3. FRONTEND -> API: The script enters the `frontend` container and curls the 
   `backend` API. Result: SUCCESS (Public access enabled).
4. FRONTEND -> DB: The script enters the `frontend` container and tries to 
   connect to the `db` container on port 27017. Result: FAILURE/TIMEOUT 
   (Isolation Verified).

23. WHY USE BRIDGE NETWORKS?
----------------------------
Bridge networks provide:
- DNS-based service discovery (No hardcoded IPs).
- Automatic container-to-container communication within the same network.
- Complete isolation from containers on other networks.
- Performance: Bridge networks are highly efficient for local container communication.

26. VIVA Q&A: DOCKER NETWORKING DEEP DIVE
-----------------------------------------
Q: What is the default network driver in Docker?
A: Bridge. It's the most common for single-host container communication.

Q: How does a container resolve the IP of another container by its name?
A: Docker has an internal DNS server that maps container names (or aliases) 
   to their current IP addresses within the custom network.

Q: Can a container be part of more than two networks?
A: Yes, a container can be connected to multiple networks, allowing it 
   to act as a gateway or bridge between them, as our backend does.

Q: Why not just use the 'host' network?
A: Host networking removes isolation between the container and the host machine, 
   posing a security risk and potential port conflicts. Bridge networks 
   provide a private, isolated virtual network.

Q: How did you verify isolation?
A: "I wrote a bash script that attempts to reach the database from the frontend 
    container. Since they are on different networks with no common bridge, 
    the request times out, proving that the database is isolated from the 
    public-facing tier."
